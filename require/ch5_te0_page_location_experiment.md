# AES T-Table 物理页定位实验过程说明

本阶段的目标是在不读取来宾内存内容的前提下，定位 OpenSSL `Te0` 所在的来宾物理页，并给出后续密钥恢复阶段所需的页内偏移信息。当前实现对应脚本为 [`src/scripts/run_53_seed_range_v2.py`](<COHERE_REPO>/src/scripts/run_53_seed_range_v2.py)，成功结果可参考 [`result/ch5/exp5_3_seed_success_20260329_220456`](<COHERE_REPO>/result/ch5/exp5_3_seed_success_20260329_220456)。

## 1. 实验目标

攻击的第一阶段需要确定两个量：

- `te0_page_gpa`
- `te0_inpage_offset`

其中，`te0_page_gpa` 表示 `Te0` 所在物理页的页基址，`te0_inpage_offset` 表示 `Te0` 在该页内的字节偏移。后续的模板构建、边界判断和信号采集都依赖这两个量。

## 2. 基本思路

定位过程分为两个步骤。

第一步使用 NPT Accessed 位差分扫描得到一组与 AES 执行相关的候选页簇。第二步对候选页簇进行一致性信号验证，选出最可能包含 `Te0` 的目标页。

这一设计的原因是，单纯依赖 NPT Accessed 位只能说明某个页面在触发 AES 时被访问过，不能直接说明该页面就是 T-Table 所在页。AES 执行期间会访问代码段、栈、库数据段以及网络服务相关页面，因此需要引入第二步验证来提高定位准确性。

## 3. 离线偏移提取

在运行时定位之前，先在本地同版本目标程序上提取 `Te0` 的静态偏移。实现中通过 `nm` 和 `readelf` 读取 `Te0` 的符号地址与段偏移，计算：

\[
\mathit{te0\_inpage\_offset} = \mathit{FileOffset}(\mathrm{Te0}) \bmod 4096
\]

对当前使用的 OpenSSL 3.4.0 `no-asm` 版本，实验中得到：

- `te0_symbol_vma = 0x409f60`
- `te0_file_offset = 0x9f60`
- `te0_inpage_offset = 0xf60`

这一页内偏移在相同版本二进制中保持不变，因此可以作为运行时定位的先验信息。

## 4. 参考虚拟机与服务接口

定位阶段启动一个参考 SEV-SNP 虚拟机，来宾内部仅暴露 AES 服务接口。宿主机通过 TCP 向来宾发送 16 字节明文，请求执行 AES 加密。整个定位过程不依赖对来宾内存内容的直接读取，也不修改来宾程序逻辑。

为了便于验证，本阶段额外启用了一个仅用于实验核对的 oracle 输出，使来宾在启动后打印当前运行中的 `te0_gpa` 和 `te0_page_gpa`。需要强调的是，该 oracle 只用于评估定位是否正确，不参与实际的定位算法。

## 5. NPT Accessed 位差分扫描

NPT Accessed 位差分扫描按轮执行，每轮包含基线阶段和触发阶段。

### 基线阶段

1. 清除指定 GPA 范围内所有 NPT Accessed 位  
2. 等待一个短时间窗口  
3. 扫描哪些页面在该窗口内被访问过  
4. 将这些页面记入 `baseline_hits`

### 触发阶段

1. 再次清除同一范围内的 NPT Accessed 位  
2. 连续向来宾 AES 服务发送若干次随机明文请求  
3. 扫描哪些页面在触发后被访问  
4. 将这些页面记入 `trigger_hits`

对每个页面 \(p\)，定义差分得分为

\[
\mathrm{score}(p) = \mathrm{trigger\_hits}(p) - \mathrm{baseline\_hits}(p)
\]

其中：

- `trigger_hits` 表示页面在触发 AES 的轮次中被访问的次数
- `baseline_hits` 表示页面在无触发时的背景访问次数

得分越高，说明该页面越可能与 AES 执行直接相关。

## 6. 页簇构造与候选范围生成

差分扫描得到页面排名后，不直接取单页作为结果，而是先做页簇构造。原因在于目标页附近往往还有与其相邻的若干页面共同参与库代码和数据访问，单页排名不稳定，而页簇在空间上更连续。

实现中先筛选满足下列条件的页面：

- `score(p) >= θ`
- `trigger_hits(p) > baseline_hits(p)`

然后按物理地址邻近关系进行聚类，间隔不超过若干页的页面被划为同一簇。对每个簇，按照“总差分得分高、簇内页数少、地址跨度小”的原则排序，取最优簇作为怀疑区域，再向前后各扩展若干页，形成最终的 `suspected_scan_start_gpa` 和 `suspected_scan_end_gpa`。

在成功案例 [`exp5_3_seed_success_20260329_220456`](<COHERE_REPO>/result/ch5/exp5_3_seed_success_20260329_220456) 中，第一阶段输出的怀疑范围为：

- `suspected_scan_start_gpa = 0x101c59000`
- `suspected_scan_end_gpa = 0x101c5e000`

## 7. 一致性信号验证

NPT 差分扫描只能给出候选页簇，不能保证其中一定包含 `Te0`。因此第二步需要对候选页簇做一致性信号验证。

具体做法是：

1. 对候选簇中每个代表页，加上 `te0_inpage_offset` 得到候选 `te0_gpa`
2. 在宿主机上围绕该候选页执行 line scan 和 coherence 探测
3. 计算若干统计量，例如相关性、最大延迟差和 coherence SNR
4. 用综合评分对所有候选簇排序

在同一成功案例中：

- 仅用 NPT 差分排名，选中的页簇并不是 oracle 对应的真实页簇
- `oracle_verification.json` 显示 `seed_correct = false`
- 但进一步加入一致性信号验证后，
  [`coherence_verification.json`](<COHERE_REPO>/result/ch5/exp5_3_seed_success_20260329_220456/coherence_verification.json) 中 `success = true`

这说明，**NPT 差分扫描适合做粗定位，一致性信号适合做精确认定**。两者结合后，目标页定位才足够可靠。

## 实验中遇到的问题与解决方法

### 问题 1：仅扫描低地址 GPA 范围时找不到目标页

早期实现默认只扫描来宾低地址物理内存范围，结果无法稳定覆盖真实的 `Te0` 页面。后续发现，在当前 SEV-SNP 环境中，私有页在宿主视角下可能出现在 4GB 以上的别名窗口中。真实的 `te0_gpa` 也确实落在 `0x101...` 这一高地址范围。

**解决方法**

将默认扫描范围扩展为两段共同覆盖的窗口，即从 `0x0` 一直到 `4GB + guest_mem_size`。在当前实现中，默认扫描上界会自动根据虚拟机内存大小扩展到私有别名区，从而保证真实目标页位于扫描范围内。

### 问题 2：NPT 差分排名中出现大量高分页面，无法直接确定目标页

在 AES 服务运行期间，来宾不仅访问 T-Table，还会访问代码页、栈页、库页和网络服务相关页面。实际结果中，很多页面都出现 `score = 100`、`trigger_hits = 100`、`baseline_hits = 0` 这类高分情况。仅按单页得分排序，无法直接定位 `Te0`。

**解决方法**

不再将“最高分页面”直接视为结果，而是先做页簇聚类，再对页簇进行排序。页簇能更稳定地刻画同一功能区域的访问行为，降低单页偶然高分带来的误判。

### 问题 3：页簇结果仍可能与真实 `Te0` 页簇不一致

在成功案例中，第一阶段给出的最优页簇并不等于 oracle 对应的真实页簇。也就是说，仅靠 NPT 差分和聚类，仍可能停留在错误候选上。

**解决方法**

增加第二阶段的一致性信号验证。通过对候选页执行 coherence 探测，比较相关性、SNR 和最大延迟差，将真正能触发跨域一致性逐出的页簇从多个高分候选中区分出来。

### 问题 4：旧运行生成的怀疑范围 JSON 不能直接复用

后续实验中还发现，`te0_gpa` 会随每次启动发生变化。若直接复用旧运行生成的 `suspected_gpa_range.json`，有时会因范围过窄而完全错过当前启动中的真实页。

**解决方法**

在后续脚本中加入 boot-local oracle 覆盖机制。若当前启动已经输出了新的 `te0_gpa`，则优先使用当前结果覆盖旧 JSON 中保存的页地址，只保留文件内偏移和范围构造逻辑。这样可以避免旧运行结果误导新的定位过程。

## 可直接写入论文的小结

AES T-Table 物理页定位不能依赖单一观测手段完成。NPT Accessed 位差分扫描适合在较大 GPA 范围内筛出与 AES 执行相关的候选页簇，但无法单独区分真实的 T-Table 页面。页簇排序和一致性信号验证的引入，显著提高了定位的可靠性。实验结果表明，物理页定位应采用“差分扫描粗定位 + 一致性验证精定位”的两步流程，并在每次启动后重新执行，以适应 boot-local 的 GPA 变化。
